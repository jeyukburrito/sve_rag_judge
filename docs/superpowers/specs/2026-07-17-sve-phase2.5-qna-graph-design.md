# SVE 2.5단계 — 카드-Q&A 관계 조인 검색 설계 문서

날짜: 2026-07-17
상태: 승인됨

## 배경

蒼炎の猛威 사건(2026-07-17): 카드 개별 재정 Q2663("양쪽 필드에 선택 가능한 팔로워가 있을 때만 플레이 가능")이 데이터에 존재하고 카드에 연결까지 되어 있었지만, 임베딩 검색이 이를 놓쳐 판정기(qwen3:8b)와 상위 모델 모두 일반 원칙 유추로 오답을 냈다. 교훈: **카드 개별 재정 > 일반 원칙 유추**이며, 카드가 식별되면 연결 재정은 임베딩이 아니라 조인으로 가져와야 한다.

데이터 관계 규모: QnA→카드 직접 연결 2,576건(카드 3,626종, 카드당 최대 29건), QnA 본문 『카드명』 언급 1,368건(91% 카드DB 매칭), 카드 효과 『토큰/카드』 참조 1,634장, 동명 재판 그룹 2,194종.

그래프 DB는 도입하지 않는다 — 노드 ~1만/간선 ~2만 규모에 질의 패턴이 고정 1홉이라 사전 계산 dict로 충분. 멀티홉·분석·시각화가 필요해지는 3단계 이후 재검토.

## 1. 관계 맵 생성 (build/graph.json)

`src/build_phase2_index.py`에 그래프 빌드 단계 추가. 임베딩 불필요, 순수 JSON, 수 초.

```json
{
  "code_to_qa":    {"BP20-063": ["Q2662", "Q2663"]},
  "name_to_codes": {"蒼炎の猛威": ["BP20-063", "PR-553"]},
  "code_to_refs":  {"BP01-001": ["薔薇の一撃"]}
}
```

- `code_to_qa`: unique_qna.jsonl의 cards 배열에서 역산
- `name_to_codes`: carddb 전체에서 이름→코드들 (동명 재판 그룹)
- `code_to_refs`: effect의 『([^』]+)』 중 **카드DB에 실재하는 이름만** 유지
- 재정 공유는 이름 단위 union: 식별된 코드 → 동명 그룹 전체 코드의 연결 재정 합집합

## 2. 질의 시 1홉 조인 (retrieve_all 확장)

순서: ① 카드 코드 정확 조회(기존) → ② 임베딩 검색(기존 쿼터 룰6/카드4/Q&A4 불변) → ③ **연결 재정 주입**: ①의 코드 + ②의 카드 상위 2종에 대해 동명 union 후 연결 QnA를 `qa_id $in` 정확 조회, 거리 0.0으로 컨텍스트 선두 배치 → ④ **토큰 참조**: ①의 코드/②의 상위 1종이 참조하는 카드 문서 포함.

컨텍스트 예산 상한(상수):
- 연결 재정: 카드당 최대 6건 **날짜 최신순**, 전체 최대 12건
- 토큰 참조 문서: 최대 2건
- 최악 총 근거 ~28건, 평균 ~18건

시작 시 graph.json 로드. 없으면 "먼저 실행하세요: python -m src.build_phase2_index" 안내 후 종료.

## 3. 프롬프트 v3

기존 우선순위 문구 뒤 한 문장 추가: "카드에 직접 연결된 Q&A 재정은 일반 원칙의 유추보다 항상 우선하며, 인용 전 각 근거의 전제 조건이 질문 상황과 일치하는지 확인하라."

## 4. 검증

- 자동(pytest, LLM/DB 불필요): 그래프 빌드 3함수(code_to_qa 역산, name_to_codes, code_to_refs 추출) + 조인 셀렉션(카드당/전체 상한, 최신순 정렬)
- **수용 게이트**: "蒼炎の猛威은 상대 필드에 아무 추종자가 없어도 사용 가능한가요?" → "플레이 불가" 취지 + [Q&A Q2663] 인용 필수. 기존 5문항 회귀 통과.
